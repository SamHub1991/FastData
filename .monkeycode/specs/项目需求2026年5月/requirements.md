# 项目需求2026年5月 - 需求文档

## 1. 背景

FastData 是一个支持 Code First、Db First、Lambda 查询、XML SQL Map、AOP 与 Repository 模式的 ORM 框架。当前需求聚焦于完善 ORM 生态工具链，包括 Model 生成工具、数据库同步工具、多数据库配置简化、中文使用文档，以及框架架构优化。

## 2. 目标

1. 完善 FastData ORM 框架能力，优化架构，抽取重复代码，降低包体积，提高扩展性、健壮性和可用性。
2. 提供 WinForms 版本的 Model 生成工具，通过可视化操作从数据库表生成 C# Model 映射代码。
3. 提供 WinForms 版本的数据同步工具，通过中间库机制实现多数据库之间的数据同步。
4. 简化多数据库配置方式，降低使用门槛。
5. 生成完整中文使用文档，覆盖 ORM 框架和配套工具。

## 3. 范围

### 3.1 本期包含

1. Model 生成工具。
2. 数据库同步工具。
3. 中间库建库脚本导出能力。
4. 中间库数据清理策略。
5. 多数据库连接与配置简化。
6. 中文文档。
7. 关键重复代码抽取与基础架构优化。

### 3.2 支持数据库

1. SQL Server。
2. MySQL。
3. Oracle。
4. 预留其他 ADO.NET 数据库扩展点。

## 4. 用户角色

1. 开发人员：使用 ORM 框架进行业务开发，并通过 Model 生成工具生成实体类。
2. 数据管理员：配置源库、目标库和同步策略，监控同步状态。
3. 运维人员：部署工具，导出建库脚本，处理同步失败和中间库清理。

## 5. 功能需求

### 5.1 Model 生成工具

#### 5.1.1 数据库选择

当用户打开 Model 生成工具时，系统应允许用户选择数据库类型，包括 SQL Server、MySQL 和 Oracle。

当用户选择数据库类型后，系统应展示该数据库类型对应的连接配置表单。

当用户输入连接信息并点击测试连接时，系统应验证连接有效性并返回明确结果。

#### 5.1.2 表选择

当数据库连接成功后，系统应加载当前数据库下的数据表列表。

当表数量较多时，系统应支持按表名关键字过滤。

当用户选择表时，系统应支持多选表。

当用户选择表后，系统应展示表字段、字段类型、主键信息、可空信息和注释信息。

#### 5.1.3 命名空间配置

当用户生成 Model 前，系统应允许用户指定默认命名空间。

当用户需要调整单个表的命名空间时，系统应允许用户编辑该表对应的命名空间。

当用户编辑命名空间时，系统应校验命名空间格式。

#### 5.1.4 生成配置

当用户选择表并设置命名空间后，系统应允许用户配置类名命名规则、属性命名规则、是否生成表特性、是否生成字段特性、是否覆盖已有文件。

当用户配置输出目录时，系统应校验目录是否可写。

当用户点击生成时，系统应为选中表生成 C# Model 文件。

#### 5.1.5 预览与可编辑

当用户生成 Model 前，系统应提供代码预览。

当用户查看预览时，系统应允许用户编辑生成内容。

当用户确认生成时，系统应以编辑后的内容写入目标文件。

#### 5.1.6 通用性

当新增数据库类型时，Model 生成工具应通过数据库元数据读取接口扩展，避免修改 WinForms 主流程。

当新增生成模板时，Model 生成工具应通过模板接口扩展，避免修改数据库读取逻辑。

### 5.2 数据库同步工具

#### 5.2.1 数据库选择

当用户打开数据同步工具时，系统应允许用户配置源数据库、中间库数据库和目标数据库。

当用户选择数据库类型后，系统应展示对应连接配置表单。

当用户点击测试连接时，系统应验证源库、中间库和目标库连接有效性。

#### 5.2.2 中间库创建

当用户配置同步任务时，系统应允许用户创建中间库结构。

当自动创建中间库失败时，系统应展示失败原因。

当用户需要手动处理时，系统应支持导出创建中间库的 SQL 语句。

当用户导出 SQL 语句时，系统应根据中间库数据库类型生成对应方言的建库脚本。

#### 5.2.3 同步流程

当同步任务运行时，系统应先将源库变更数据写入中间库。

当中间库接收到待同步数据后，系统应根据同步策略将数据写入目标库。

当目标库写入成功后，系统应更新中间库同步状态。

当目标库写入失败时，系统应记录错误信息，并根据重试策略处理。

#### 5.2.4 同步策略

当用户配置同步任务时，系统应支持全量同步、增量同步和定时同步。

当用户配置实时同步时，系统应尽量缩短源库到中间库、中间库到目标库的处理延迟。

当数据同步失败时，系统应支持重试次数、重试间隔和失败挂起策略。

当目标表存在冲突数据时，系统应支持插入、更新、覆盖、跳过和自定义处理策略。

#### 5.2.5 中间库清理

当中间库数据超过保留时间时，系统应自动清理已完成同步的数据。

当中间库数据超过容量阈值时，系统应按策略清理可安全删除的数据。

当存在失败或待处理数据时，系统应保留必要数据用于排查和重试。

当清理任务执行时，系统应记录清理数量、清理范围和执行结果。

#### 5.2.6 稳定性

当同步工具异常退出后，系统应支持从中间库状态恢复任务。

当网络或数据库短暂不可用时，系统应通过重试机制保持任务可恢复。

当同步任务运行时，系统应记录运行日志、错误日志和关键状态。

#### 5.2.7 通用性

当新增源数据库或目标数据库类型时，系统应通过数据库适配器扩展。

当新增同步策略时，系统应通过策略接口扩展。

当新增中间库方言时，系统应通过脚本生成器扩展。

### 5.3 多数据库配置简化

当用户配置多个数据库连接时，系统应提供统一配置结构，减少 Oracle、MySQL 和 SQL Server 配置重复。

当用户设置默认数据库时，系统应允许业务调用省略数据库 Key。

当用户需要按数据库类型扩展参数时，系统应保留数据库特定配置项。

当配置缺失或格式错误时，系统应输出明确错误信息。

### 5.4 中文使用文档

当功能完成后，系统应提供中文 README 或 docs 文档。

文档应覆盖 ORM 快速开始、数据库配置、Code First、Db First、XML SQL Map、Repository、AOP、Model 生成工具、数据同步工具和常见问题。

文档中的命令示例应可直接复制执行。

### 5.5 架构优化

当发现不同数据库实现存在重复代码时，系统应抽取公共接口和公共基类。

当核心功能需要扩展数据库类型时，系统应通过适配器模式扩展。

当工具代码与 ORM 核心代码交互时，系统应优先复用 FastData 已有能力。

当调整架构时，系统应保持现有公开 API 的可用性。

## 6. 非功能需求

### 6.1 可用性

工具界面应提供清晰操作流程、连接测试、生成预览、错误提示和操作日志。

### 6.2 实时性

数据同步工具应支持秒级轮询或更低延迟的同步调度策略，具体延迟目标应可配置。

### 6.3 稳定性

数据同步工具应保证同步状态可追踪、失败可重试、异常可恢复。

### 6.4 扩展性

数据库访问、元数据读取、代码生成模板、同步策略、清理策略和 SQL 脚本生成应具备接口化扩展点。

### 6.5 兼容性

工具应优先兼容当前 FastData 的 .NET Framework 版本和现有项目结构。

### 6.6 安全性

数据库连接字符串应避免明文展示，配置文件保存时应支持脱敏展示和可选加密存储。

## 7. 验收标准

1. Model 生成工具可以连接 SQL Server、MySQL、Oracle 中至少一种数据库并生成可编译的 Model 文件。
2. Model 生成工具支持多表选择、命名空间指定、命名空间编辑、代码预览和输出目录选择。
3. 数据库同步工具可以完成源库到中间库、中间库到目标库的同步流程。
4. 数据库同步工具可以导出中间库建库 SQL。
5. 数据库同步工具具备失败重试、状态记录和中间库自动清理能力。
6. 多数据库配置结构比现有 XML 配置更简洁，并保持兼容现有调用方式。
7. 中文使用文档覆盖核心框架与两个工具的使用流程。
8. 核心架构优化后，重复数据库处理逻辑减少，新增数据库类型具备清晰扩展路径。
